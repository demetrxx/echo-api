- ограничиваешь:
  - макс 2–3 chunks на 1 article (чтобы не зацикливаться)
  - разнообразие источников (source diversity)
- формируешь “context bundle” для генерации

## 1) База: таблица сущностей + уникальный ключ

### `entities`
- `id` (uuid или text) — внутренний идентификатор
- `type` (org/person/product/tech/topic…)
- `canonical_name` (нормализованное имя)
- `display_name` (как показывать)
- `created_at`

**Уникальность:**
- `UNIQUE(type, canonical_name)`

Это уже гарантирует: **дубликаты не создадутся** при конкурирующих запросах.

---

## 2) Как делать `canonical_name` (нормализация)

Делаешь функцию `canonicalize(name)`:

- lowercase
- trim
- заменить множественные пробелы на один
- убрать кавычки/скобки/лишнюю пунктуацию
- опционально: убрать “inc”, “ltd”, “corp” для `org`
- опционально: Unicode normalize (NFKC)

Примеры:
- `" OpenAI, Inc. "` → `openai`
- `"GPT-4"` → `gpt-4`
- `"Sam Altman"` → `sam altman`

Важно: **canonical_name должен быть детерминированным**.

## 3) Как создавать `id`

### Лучший практичный вариант:
- `id = uuid` (или ulid) **как первичный ключ**
- `canonical_name` + `type` отвечают за дедуп

Почему это лучше, чем “id = slug”:
- slug меняется (переименования, исправления)
- uuid стабильнее
- алиасы проще

Но тебе хочется “красивый id” для API? Тогда:

### Компромисс:
- `id` = `"type:slug"` **только после дедупа**
- внутри всё равно держи `entity_pk uuid`

Я бы сделал так:
- PK: `entity_id` (uuid)
- Public ID: `entity_key` (text) = `type + ":" + slug(canonical_name)` с `UNIQUE(entity_key)`

## 4) Как не плодить “OpenAI” vs “Open AI” vs “ChatGPT” vs “GPT”

Тут нужны **алиасы**.

### `entity_aliases`
- `alias_canonical` (нормализованная строка)
- `entity_id` (FK на entities)
- `source` (llm/user/system)
- `created_at`

**Уникальность:**
- `UNIQUE(alias_canonical, type)` (или просто alias_canonical если тип хранится рядом)

Флоу:
1) Ты извлёк `"Open AI"` как org
2) canonicalize → `open ai`
3) сначала ищешь в `entity_aliases` (alias → entity_id)
4) если не найдено — ищешь в `entities` по `(type, canonical_name)`
5) если не найдено — создаёшь entity и сразу пишешь alias

Так ты постепенно обучаешь систему синонимам.

---

## 5) Реальный алгоритм “upsert entity” (надёжный)

1) `canonical = canonicalize(name)`
2) `entity_id = select entity_id from entity_aliases where type=? and alias_canonical=?`
3) если нет:
   - `entity_id = select id from entities where type=? and canonical_name=?`
4) если нет:
   - `insert into entities(type, canonical_name, display_name) values (...) on conflict(type, canonical_name) do update set ... returning id`
5) `insert into entity_aliases(type, alias_canonical, entity_id) on conflict do nothing`

Эта схема выдерживает параллельные запросы и не создаёт дублей.

## Что считается “хорошим” чанкованием в твоём продукте

- Чанк должен быть **самодостаточным** (не обрывок одной фразы).
- Должен быть **не слишком длинным** (чтобы retrieval попадал точно).
- Должен сохранять **структуру** (заголовки/буллеты не разваливаются).
- Нужен **детерминизм** (один и тот же текст → те же чанки).
- Быть дешёвым и быстрым.

---

## Подход 1 (самый лёгкий): фиксированное окно по символам/словам
**Как:** режешь каждые N символов/слов + overlap.  
**Плюсы:** 5 минут кода.  
**Минусы:** ломает предложения, заголовки, списки; хуже качество.

**Не рекомендую** для quality-first продукта.

---

## Подход 2 (лучший лёгкий): структура → склейка до лимита → overlap
**Это мой рекомендованный вариант.**

### Шаги:
1) **Нормализуешь текст**  
- убираешь мусор (навигацию, футер, “подпишитесь”)
- конвертишь в “параграфы” (split по пустым строкам)

2) **Сегментация по структуре**
- если есть Markdown/HTML: учитываешь `h1-h3`, `li`, `p`, `blockquote`
- иначе: простой “параграфный” режим

3) **Склейка (packing)**
- берёшь сегменты и склеиваешь, пока не достигнешь `target_size`
- если сегмент слишком длинный — внутри него делишь по предложениям

4) **Overlap**
- добавляешь небольшой overlap (например 10–15% чанка) для “сшивки смысла”
- overlap лучше делать **по предложениям**, а не по символам

### Рекомендуемые размеры (старт):
- `target`: ~300–450 токенов
- `max`: ~600–700 токенов
- `min`: ~120–180 токенов
- overlap: ~50–80 токенов (или 1–2 предложения)

Почему такие:
- достаточно для смысла
- достаточно коротко для точного retrieval
- не слишком много чанков (стоимость эмбеддингов)

## Подход 3 (слегка умнее): sentence-aware + “semantic paragraph”

А пайплайн делает всё остальное:

```ts
class IngestPipeline {
  constructor(
    private normalizer: Normalizer,
    private chunker: ChunkerRouter,
    private deduper: Deduper
  ) {}

  async ingest(adapter: SourceAdapter, uri: string) {
    const items = await adapter.fetch(uri);

    const articles: Article[] = [];
    for (const item of items) {
      const article = this.normalizer.toArticle(adapter.type, uri, item);

      const existing = await this.deduper.findExisting(article);
      const finalArticle = existing ?? await this.deduper.insertArticle(article);

      const chunks = this.chunker.chunk(finalArticle);
      await this.deduper.upsertChunks(finalArticle.id!, chunks);

      articles.push(finalArticle);
    }
    return articles;
  }
}
```

Это единственная точка, которая “инкорпорирует” всё.

```ts
export interface IngestRequest {
  sourceType: SourceType;
  sourceUri: string;
  params?: FetchParams;
}

export class IngestPipeline {
  constructor(
    private adapters: AdapterRegistry,
    private normalizers: NormalizerRegistry,
    private common: CommonNormalizer,
    private chunker: ChunkerRouter,
    private store: ArticleStore,
    private embedder?: Embedder,
    private embeddingStore?: EmbeddingStore,
  ) {}

  async run(req: IngestRequest): Promise<{ articles: Article[]; chunkCount: number }> {
    const adapter = this.adapters.get(req.sourceType);
    const platformNorm = this.normalizers.get(req.sourceType);

    const rawItems = await adapter.fetch(req.sourceUri, req.params);

    const articles: Article[] = [];
    let chunkCount = 0;

    for (const raw of rawItems) {
      const doc = await platformNorm.normalize(raw);
      const article = this.common.toArticle(req.sourceUri, doc);

      // upsert article
      const upserted = await this.store.upsertArticle(article);
      const saved = upserted.entity;

      // chunk
      const chunks = this.chunker.chunk(saved);
      await this.store.upsertChunks(saved.id!, chunks);
      chunkCount += chunks.length;

      // embeddings (если включено)
      if (this.embedder && this.embeddingStore) {
        // тут важно: эмбедить только новые/изменившиеся чанки (по hash)
        // реализация — внутри store.upsertChunks может вернуть список новых chunkId
      }

      articles.push(saved);
    }

    return { articles, chunkCount };
  }
}
```

## 4) Как это всё “сцепляется” (в 6 строк)

- `IngestPipeline.run()`  
  → берёт `SourceAdapter.fetch()`  
  → для каждого `RawItem` делает `PlatformNormalizer.normalize()`  
  → потом `CommonNormalizer.toArticle()`  
  → `ArticleStore.upsertArticle()`  
  → `ChunkerRouter.chunk()`  
  → `ArticleStore.upsertChunks()`  
  → (опционально) `Embedder.embedChunks()` + `EmbeddingStore.upsertEmbeddings()`

## 2) Практически: YouTubeNormalizer как “мини-роутер”

```ts
class YoutubeNormalizer extends PlatformNormalizer {
  readonly sourceType: SourceType = 'youtube';

  async normalize(raw: RawItem): Promise<PlatformDoc> {
    const kind = raw.rawMeta?.ytKind as 'video' | 'community' | undefined;

    if (kind === 'community') return this.normalizeCommunity(raw);
    // default = video
    return this.normalizeVideo(raw);
  }

  private async normalizeCommunity(raw: RawItem): Promise<PlatformDoc> {
    const text = raw.rawText ?? '';
    const blocks: Block[] = [
      // можно добавить heading с названием канала/типа, но необязательно
      ...textToParagraphBlocks(text),
      ...attachmentsToBlocks(raw.rawMeta),
    ];

    return {
      sourceType: 'youtube',
      externalId: raw.externalId,             // postId
      canonicalUri: raw.uri,
      title: raw.title ?? 'YouTube community post',
      publishedAt: raw.publishedAt,
      author: raw.author,
      contentType: 'post',
      blocks,
      rawMeta: raw.rawMeta,
    };
  }

  private async normalizeVideo(raw: RawItem): Promise<PlatformDoc> {
    // Вариант 1: транскрипт уже лежит в rawMeta/rawText
    // Вариант 2: нормализатор сам тянет транскрипт (дороже и хуже для архитектуры)
    const transcript = raw.rawMeta?.transcriptText as string | undefined;
    const description = raw.rawMeta?.description as string | undefined;

    const blocks: Block[] = [];

    // Лёгкая структура (без финальной чистки)
    if (description) {
      blocks.push({ kind: 'heading', level: 2, text: 'Description' });
      blocks.push(...textToParagraphBlocks(description));
    }

    if (transcript) {
      blocks.push({ kind: 'heading', level: 2, text: 'Transcript' });
      blocks.push(...transcriptToBlocks(transcript, raw.rawMeta));
    } else {
      // если транскрипта нет — всё равно полезно иметь “контент” из title/description
      blocks.push({ kind: 'heading', level: 2, text: 'Content' });
      blocks.push(...textToParagraphBlocks(raw.title ?? ''));
      if (!description) {
        // крайний случай: хоть что-то
        blocks.push(...textToParagraphBlocks(raw.rawText ?? ''));
      }
    }

    return {
      sourceType: 'youtube',
      externalId: raw.externalId,            // videoId
      canonicalUri: raw.uri,
      title: raw.title ?? 'YouTube video',
      publishedAt: raw.publishedAt,
      author: raw.author,
      contentType: transcript ? 'transcript' : 'article',
      blocks,
      rawMeta: raw.rawMeta,
    };
  }
}
```

Ключ: **нормализатор не “чистит до идеала”**, он строит блоки и корректно выбирает `contentType`.

## 5. Конкретные механизмы, которые решают твою проблему

### 5.1 Neighbor expansion (обязательно)

Если чанк `i` попал в top-k:
- автоматически подтягиваешь:
  - `i-1`
  - `i+1`

(если они есть)

Это:
- восстанавливает ход мысли
- почти бесплатно
- **резко** улучшает качество

> Это must-have. Без этого ты действительно будешь терять идеи.

---

### 5.2 Structural expansion (очень сильно)

Если у чанка есть `headingPath`:
- подтягиваешь другие чанки под тем же заголовком
- или хотя бы их часть

Это возвращает “идею раздела”, а не абзаца.

---

### 5.3 Rerank уже на расширенном пакете

Сначала:
- вектор → якоря

Потом:
- rerank на:
  - чанке
  - чанке+соседях
  - или собранном пакете

Реранкер (даже простой LLM judge) умеет:
- понимать “идею”
- понимать “слабые, но важные” фрагменты

### 5.4 Diversity constraints (против туннеля)

Если одна статья дала 3 топ-чанка:
- **ограничиваешь**
- берёшь 1–2
- берёшь другие источники

Это:
- спасает от “одна идея = один автор”
- даёт более богатый контекст

## 1. Ты задал правильный вопрос, потому что…

Ты описываешь **long-form reasoning**:

- мысль развивается
- аргумент строится постепенно
- вывод опирается на цепочку
- каждый отдельный абзац слаб, но вместе — сильны

Это:
- эссе
- аналитика
- философия
- сложные экспертные рассуждения

**И да — такой смысл нельзя восстановить “чанк + 1 сосед” по умолчанию.**

---

## 2. Почему фиксированное правило “±1 чанк” — недостаточно

### Представь структуру:

```
[Setup] → [Context] → [Assumption]
→ [Reasoning step 1]
→ [Reasoning step 2]
→ [Counterargument]
→ [Conclusion]
```

Если embedding вытащил:
- `[Reasoning step 2]`

То:
- `i-1` = step 1 (хорошо)
- `i+1` = counterargument (частично)
- **но setup и conclusion потеряны**

👉 Получается “кусок логики без основания и вывода”.

---

## 3. Ключевой сдвиг: **не “сколько чанков”, а “пока смысл не замкнулся”**

Правильная модель:

> **Мы расширяем контекст до тех пор, пока рассуждение не стало самодостаточным.**

Это не число. Это условие.

---

## 4. Практическая стратегия: 3 уровня расширения (quality-first)

### Уровень 1 — Минимальный (по умолчанию)
- anchor chunk
- ±1 сосед

**Дёшево, быстро, хорошо для новостей, фактов, коротких мыслей.**

---

### Уровень 2 — Структурный (если мысль “длинная”)

Триггеры:
- чанк длиннее среднего
- нет явного утверждения/вывода
- много логических связок: *however, therefore, because, but, on the other hand*
- высокая rerank relevance, но низкая “self-containedness”

Действия:
- расширение до **границ секции** (по `headingPath`)
- или ±2–3 чанка

👉 Это уже восстанавливает ход рассуждения.

---

### Уровень 3 — Semantic closure (редко, но критично)

Используется, когда:
- тема философская / аналитическая
- angle требует глубины
- генерируется opinion-piece

Как:
- итеративно добавляешь чанки
- и проверяешь:
  > “Можно ли понять мысль без внешнего контекста?”

Это можно делать:
- эвристиками
- или лёгким LLM-чекером (“is this reasoning complete?”)

👉 В итоге ты получаешь **reasoning span**, а не “набор чанков”.

---

## 5. Важный момент: **retrieval ≠ assembly**

Ты не обязан:
- эмбедить большие куски
- искать по большим кускам

Ты обязан:
- **искать по якорям**
- **собирать по логике**

Это фундаментально разные этапы.

---

## 6. Почему это не убивает производительность (важно)

- Anchor retrieval остаётся дешёвым
- Расширение — локальное (по article_id)
- Rerank — на десятках чанков, не тысячах

То есть:
- сложность контролируема
- качество растёт нелинейно

---

## 7. Как это связано с Theme и Angle (ключевая связка)

Вот тут магия Echo.

### Theme говорит:
> “О чём вообще речь?”

### Angle говорит:
> “Какую мысль я хочу провести?”

👉 **Angle определяет, нужна ли глубина.**

- “Новости LLM” → уровень 1
- “Почему хайп вокруг LLM вреден” → уровень 2–3
- “Философия агентности” → почти всегда уровень 3

Это позволяет:
- не тащить лишний контекст
- но не терять сложные идеи

## 1. Почему `Article` — плохое имя концептуально (ты это чувствуешь не зря)

Если ты:

## 1) SQL DDL

### enums
```sql
CREATE TYPE source_type AS ENUM ('tg','youtube','rss','website','twitter');
CREATE TYPE content_type AS ENUM ('post','thread','article','transcript');
CREATE TYPE assigned_by AS ENUM ('user','smart','retrain');
CREATE TYPE platform_type AS ENUM ('tg','x','linkedin','instagram');
```

### sources
```sql
CREATE TABLE sources (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL,
  type source_type NOT NULL,
  uri text NOT NULL,
  title text,
  is_active boolean NOT NULL DEFAULT true,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (user_id, type, uri)
);
CREATE INDEX sources_user_idx ON sources(user_id);
```

### documents (бывшие articles)
```sql
CREATE TABLE documents (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL,
  source_id uuid NOT NULL REFERENCES sources(id) ON DELETE CASCADE,

  source_type source_type NOT NULL,
  content_type content_type NOT NULL,

  external_id text NOT NULL,          -- msgId/videoId/rss guid etc (в рамках source_type + content subtype)
  canonical_uri text NOT NULL,        -- canonical url/id материала
  title text NOT NULL,
  author text,
  published_at timestamptz,

  text text NOT NULL,                 -- canonical cleaned text (plain/markdown-ish)
  content_hash text NOT NULL,         -- hash(text + title + content_type) детерминированный

  raw_meta jsonb,

  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

-- Идемпотентность. Минимально надёжно:
-- 1) уникальность по (source_type, external_id) часто работает для tg/youtube/rss/twitter
CREATE UNIQUE INDEX documents_source_external_uq
  ON documents(source_type, external_id);

-- 2) canonical_uri тоже полезен, но не всегда стабилен => можно сделать частично или как non-unique индекс
CREATE INDEX documents_canonical_idx ON documents(canonical_uri);

CREATE INDEX documents_user_idx ON documents(user_id);
CREATE INDEX documents_source_idx ON documents(source_id);
CREATE INDEX documents_published_idx ON documents(published_at);
```

> Если боишься, что `external_id` может пересекаться между “youtube video” и “youtube community”, добавь `external_subtype` (например `yt_kind`) в documents и включи его в unique индекс.

### document_chunks
```sql
CREATE TABLE document_chunks (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  document_id uuid NOT NULL REFERENCES documents(id) ON DELETE CASCADE,

  idx int NOT NULL,
  text text NOT NULL,

  start_char int NOT NULL,
  end_char int NOT NULL,

  heading_path text,
  hash text NOT NULL,                 -- hash(text) для дедупа/переэмбеда

  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),

  UNIQUE (document_id, idx)
);

CREATE INDEX chunks_document_idx ON document_chunks(document_id);
CREATE INDEX chunks_hash_idx ON document_chunks(hash);
```

### chunk_embeddings
```sql
CREATE TABLE chunk_embeddings (
  chunk_id uuid NOT NULL REFERENCES document_chunks(id) ON DELETE CASCADE,
  model text NOT NULL,
  dim int NOT NULL,
  vector real[] NOT NULL,             -- если без pgvector. Если есть pgvector — лучше vector(1536)
  created_at timestamptz NOT NULL DEFAULT now(),
  PRIMARY KEY (chunk_id, model)
);
```

> Если у тебя есть pgvector — делай `vector vector(1536)` + индекс ivfflat/hnsw. Иначе пока можно real[] и вынести поиск наружу/в отдельное хранилище.

```sql
  post_id uuid NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  chunk_id uuid NOT NULL REFERENCES document_chunks(id) ON DELETE CASCADE,

  rank int NOT NULL,
  score_before real,
  score_after real,
  used_as text,                       -- optional: 'fact'|'quote'|'inspiration'

  created_at timestamptz NOT NULL DEFAULT now(),

  PRIMARY KEY (post_id, chunk_id)
);

CREATE INDEX post_contexts_post_idx ON post_contexts(post_id);
CREATE INDEX post_contexts_chunk_idx ON post_contexts(chunk_id);
```

## 1. Зачем вообще нужна `post_contexts`

Коротко:

> **Это журнал того, из какого именно знания был написан конкретный пост.**

Без неё:
- ты не знаешь, *почему* модель сказала то, что сказала
- ты не можешь дебажить галлюцинации
- ты не можешь обучать систему от реальных правок пользователя
- ты не можешь объяснить пользователю “почему этот пост выглядит так”

С ней:
- система становится **объяснимой**
- темы реально “учатся”
- Angle перестаёт быть просто строкой

---

## 2. Концептуально: что это за связь

### Не путай:

- `theme_contexts` → **что вообще релевантно теме**
- `post_contexts` → **что конкретно использовалось сейчас**

`theme_contexts` — это knowledge base темы  
`post_contexts` — это execution trace генерации

---

## 3. Как формируется `post_contexts` (по шагам)

1) Пользователь нажал **Generate post**
2) Система:
   - сформировала **query** (Theme + Angle + intent)
   - сделала retrieval (anchors)
   - расширила контекст (neighbors / sections)
   - rerank
3) Получился финальный **context pack** (набор чанков)
4) Именно **этот набор** пошёл в prompt

👉 `post_contexts` фиксирует этот набор.

---

## 4. Что значит каждая колонка

```sql
post_id uuid
```
Какому посту принадлежит этот контекст.

---

```sql
chunk_id uuid
```
Какой конкретно фрагмент знания был использован.

Это ссылка на **document_chunks**, а не documents — принципиально.

---

```sql
rank int
```
Порядок важности/использования в финальном context pack.

- `rank = 1` → самый важный
- помогает:
  - понимать, что реально влияло
  - строить explain UI (“основано в первую очередь на …”)

---

```sql
score_before real
```
Релевантность **до** rerank (например cosine similarity / bm25).

Нужно для:
- дебага retrieval
- понимания, что выбрасывается реранкером

---

```sql
score_after real
```
Релевантность **после** rerank.

Нужно для:
- обучения реранкера
- статистики качества
- анализа “слабых, но важных” фрагментов

---

```sql
used_as text (optional)
```
Роль чанка в посте:
- `fact`
- `quote`
- `argument`
- `inspiration`

Не обязательно сейчас, но:
- очень полезно позже для UI и обучения
- можно заполнить автоматически или оставить пустым

---

```sql
created_at
```
Когда именно использован.

## 5. Что это даёт тебе на практике (реальная ценность)

### 5.1 Explainability (доверие)

Ты можешь показать пользователю:
> “Этот пост основан на этих источниках”

Не абстрактно, а конкретно:
- статья X
- фрагмент Y

Это **редко у конкурентов** и очень сильно повышает доверие.

---

### 5.2 Debug качества

Ты видишь:
- какие чанки попадают в посты
- какие часто используются
- какие приводят к плохим текстам

Это позволяет:
- чистить источники
- пересчитывать relevance
- улучшать chunking

---

### 5.3 Обучение Themes и Angles

Если пользователь:
- отредактировал пост
- сказал “не похоже на меня”

Ты смотришь:
- какие чанки участвовали
- какие из них “плохие”

И:
- понижаешь их вес в `theme_contexts`
- или вообще исключаешь

👉 Это **обучение без ручной разметки**.

---

### 5.4 Персонализация профайла

Если пост с Angle X:
- использовал чанки A/B/C
- и пользователь доволен

Ты усиливаешь связь:
- Theme + Angle → эти чанки

## 6. Почему `post_contexts` НЕ избыточна

Может возникнуть мысль:
> “Но ведь контекст уже был передан в LLM, зачем хранить?”

Ответ:
- LLM — чёрный ящик
- ты должен хранить **decision trace**, а не надеяться на повторимость

Это как логи в backend:
- прод может работать и без логов
- но ты не захочешь жить без них
